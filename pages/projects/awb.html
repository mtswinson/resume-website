Advisor Workbench

Context

Internal tool consisting of dynamic user interfaces to help customer support teams efficiently manage customer information, incorporating responsive layouts, conditional logic, and data validation, while adhering to test-driven development and maintaining a goal of performance optimization.

Problem/Goal

General Motors required a reliable interface for advisors to view, update, and create customer and vehicle records while engaging with customers and third-party services. Our objective was to build the necessary components and backend services to support this workflow and improve the overall efficiency of advisor operations.

Process

I contributed to both frontend and backend development, focusing on form behavior, data validation, and integration with existing systems. The stack included React for the UI and API's for data persistence. Constraints included strict formatting rules for regional data and upstream validation requirements.

Feature Breakdown

  • Profile Details
    • This component served as the initial entry point for customer data. I built a structured form using React, designed to capture essential customer information including name, contact details, address, etc. While the logic was minimal, the form architecture emphasized clean separation of concerns—each input field was modularized for reuse and validation. On the backend, I extended the data model and updated the API layer to accept and persist the new payload structure, ensuring compatibility with upstream services.
  • Member Profile
    • A more complex form system that supported both creation and editing of customer profiles. I implemented conditional rendering and dynamic field states based on user input and existing data. For example, fields would auto-disable if certain criteria were met, and validation rules were region-specific—phone numbers, zip codes, and addresses were formatted and validated according to locale. I also introduced input masking and auto-formatting logic to reduce advisor error and improve data integrity. This component required tight coordination between frontend logic and backend validation schemas.
  • Preferred Dealer
    • This form allowed advisors to view and update a customer's preferred dealership for each vehicle. The component was designed to be context-aware: it would dynamically enable or restrict editing based on customer preferences and account status. I built a state management layer that listened to changes in customer data and adjusted the UI accordingly. The form also included a lookup feature for available dealers, integrating with a backend service to fetch and filter results. This ensured advisors could only select valid options, reducing the risk of mismatched data and improving workflow reliability

Impact/Differentiator

The tool improved advisor efficiency by consolidating key customer and vehicle data into a single interface. A phone number formatting solution I developed was adopted across multiple components, replacing a less reliable implementation and improving data consistency.

Reflection/Key Takeaway

This project reinforced the importance of thoughtful form design and backend coordination. I gained experience in building reactive interfaces that respect business rules and learned how small improvements—like better formatting logic—can have wide-reaching effects across a system.
