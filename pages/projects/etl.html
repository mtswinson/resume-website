Data Migration

Context

Participated in a large-scale data migration project, using IBM DataStage and SQL to extract, transform, and load sensitive vehicle, ownership, and coverage data from multiple legacy systems into a more mordernized and reliable system

Problem/Goal

Legacy systems were being phased out, requiring a complete transfer of customer and vehicle data to newer platforms. Migrating this data was essential to improve system reliability and ensure that advisors had access to accurate, up-to-date informationâ€”ultimately enhancing the customer experience.

Process

As someone new to DataStage and the broader ETL ecosystem, I began by familiarizing myself with the tools and workflows. I started with small-scale payload transfers in test environments, using SQL to query both source and target databases to verify data integrity. This hands-on approach helped me build confidence and understand the nuances of the migration process, including schema alignment, transformation logic, and delivery validation.

Feature Breakdown

  • Data Extraction
    • I was responsible for querying and retrieving structured data from legacy systems, which included customer profiles, vehicle ownership records, and coverage details. This involved writing SQL queries tailored to each legacy schema and ensuring that sensitive fields were handled correctly during extraction.
  • Data Transformation
    • Once extracted, the data needed to be reshaped to fit the target schema. I implemented transformation logic within DataStage to normalize formats, convert field types, and apply business rules. This included tasks like reformatting date fields, mapping legacy codes to new identifiers, and ensuring referential integrity across related tables.
  • Data Loading
    • Transformed data was loaded into the new system using DataStage pipelines. I monitored pipeline execution, handled error logging, and validated row counts and field-level accuracy post-load. I also coordinated with QA to ensure that the migrated data met acceptance criteria before being marked as production-ready.
  • Pipeline Monitoring and Delivery Validation
    • I tracked the status of data transfers, monitored server health during migration windows, and performed sanity checks on delivery batches. This included verifying that no records were dropped, truncated, or misaligned during the ETL process.

Impact/Differentiator

The migration enabled advisors to access cleaner, more reliable customer and vehicle data through modern systems. The improved data quality and system responsiveness directly supported better service outcomes. My transformation logic and validation routines helped reduce downstream errors and ensured a smoother transition from legacy infrastructure.

Reflection/Key Takeaway

This was my first experience working in an enterprise-scale ETL environment. It gave me a clear view of how data flows through backend systems and how foundational it is to application reliability. I learned how to navigate unfamiliar tools, adapt quickly, and contribute meaningfully to a high-stakes migration effort
